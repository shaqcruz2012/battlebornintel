OPTIMIZATION CYCLES OVERVIEW
================================================================================

CYCLE 1: Companies Filter Indexes (DIFFICULTY: 2/5)
  BOTTLENECK: Missing indexes on stage, region, sectors columns
  FIX: Create 7 specialized B-tree and GIN indexes
  LATENCY IMPACT: 120ms → 20ms (88% improvement)
  STORAGE: ~2.5MB
  IMPLEMENTATION: 11 CREATE INDEX statements in 009_optimization_indexes_companies.sql

CYCLE 2: Materialized Latest Scores (DIFFICULTY: 2/5)
  BOTTLENECK: Expensive CTE with DISTINCT ON on every query
  FIX: Materialize latest scores view, refresh hourly
  LATENCY IMPACT: 40-60ms CTE time → 2-3ms join (95% improvement)
  STORAGE: ~350KB
  IMPLEMENTATION: Materialized view + hourly refresh job in 010_optimization_materialized_scores.sql

CYCLE 3: Graph Edges Event Year Indexes (DIFFICULTY: 1/5)
  BOTTLENECK: No index on event_year, complex WHERE with NULLs
  FIX: Create 6 partial B-tree indexes for temporal filtering
  LATENCY IMPACT: 8-12ms → 1-2ms per query (85% improvement)
  STORAGE: ~256KB
  IMPLEMENTATION: Partial indexes in 011_optimization_indexes_graph.sql

CYCLE 4: Precomputed KPI Aggregates (DIFFICULTY: 3/5)
  BOTTLENECK: Full table scans of companies/funds + JS aggregation
  FIX: Cache computed KPIs in table, refresh every 15 minutes
  LATENCY IMPACT: 150-200ms → 5-8ms (96% improvement)
  STORAGE: ~2KB (very small)
  IMPLEMENTATION: Cache table + computation job in 012_optimization_kpi_cache.sql

CYCLE 5: Full-Text Search Indexes (DIFFICULTY: 3/5)
  BOTTLENECK: Three ILIKE conditions with leading wildcards
  FIX: Create trigram GIN indexes for fuzzy matching
  LATENCY IMPACT: 30-50ms → 5-10ms (80% improvement)
  STORAGE: ~5MB for trigram indexes
  IMPLEMENTATION: GIN indexes in 009_optimization_indexes_companies.sql (included with Cycle 1)

CYCLE 6: Denormalized Graph View (DIFFICULTY: 3/5)
  BOTTLENECK: 7 parallel queries for graph entities
  FIX: Create denormalized materialized view combining all entities
  LATENCY IMPACT: 60-80ms → 15-20ms (75% improvement)
  STORAGE: ~1.2MB (view + indexes)
  IMPLEMENTATION: Materialized view in 013_optimization_denormalized_graph.sql

CYCLE 7: Redis Query Caching (DIFFICULTY: 3/5)
  BOTTLENECK: Repeated expensive queries, no client-side caching
  FIX: Add Redis layer with TTL and event-driven invalidation
  LATENCY IMPACT: 50-100ms → 5-10ms on hits (90% improvement, peak load dependent)
  STORAGE: ~128MB Redis (configurable)
  IMPLEMENTATION: Redis client + cache layer in 014_optimization_query_caching.sql + Node.js code

RISK ASSESSMENT
================================================================================

PHASE 1A (Indexes): VERY LOW RISK
  - Indexes are read-only additions
  - No schema changes
  - No application code changes needed
  - Can be rolled back by dropping indexes
  - Concurrent creation: no table locks
  - Estimated downtime: 0 minutes

PHASE 1B (Views): LOW RISK
  - New materialized views don't affect existing queries
  - Requires refresh job, but job can be disabled
  - Can be rolled back by dropping views
  - Concurrent refresh: no table locks
  - Estimated downtime: 0 minutes
  - Application change: 1 file (additive only)

PHASE 2 (KPI Cache): LOW RISK
  - New cache table doesn't affect existing data
  - Queries have fallback to direct computation
  - Can disable by removing initialization
  - Application change: 1 file (additive only)
  - Estimated downtime: 0 minutes

PHASE 3 (Redis): MEDIUM RISK
  - Requires additional infrastructure (Redis)
  - Can be disabled via CACHE_ENABLED env var
  - If Redis fails, system falls back to direct DB queries
  - Application changes: 3 files (mostly additive)
  - Estimated downtime: 0 minutes
  - Mitigation: Start with CACHE_ENABLED=false, enable after testing

ROLLBACK PROCEDURES:
  - Phase 1A: DROP INDEX ... (immediate, 1 minute)
  - Phase 1B: DROP MATERIALIZED VIEW ... (immediate, 30 seconds)
  - Phase 2: DELETE FROM kpi_cache; (data only, no schema impact)
  - Phase 3: docker-compose down redis; set CACHE_ENABLED=false (10 seconds)
  - Full rollback: Restore from pg_dump backup (5-10 minutes)

SUPPORT AND TROUBLESHOOTING
================================================================================

If indexes not showing up:
  - Verify migration ran without errors
  - Check: SELECT * FROM pg_indexes WHERE tablename = 'companies';
  - If missing, re-run migration or check permissions

If materialized views stale:
  - Check: SELECT MAX(computed_at) FROM latest_company_scores;
  - Manual refresh: REFRESH MATERIALIZED VIEW CONCURRENTLY latest_company_scores;
  - Check refresh job is running

If Redis not working:
  - Set CACHE_ENABLED=false and restart (fallback to direct DB)
  - Check: redis-cli ping
  - View logs: docker logs <redis-container>

If performance not improved:
  - Run: ANALYZE companies; (update stats)
  - Check execution plans: EXPLAIN ANALYZE SELECT ...
  - Verify indexes are being used (see plans)
  - Review slow query log
